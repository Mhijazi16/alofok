**format.ts** — Extract the formatting functions that are duplicated everywhere:
```typescript
export function formatCurrency(val: number): string {
  return val.toLocaleString("en-US", { minimumFractionDigits: 2, maximumFractionDigits: 2 });
}

export function formatDate(dateStr: string): string {
  return new Date(dateStr).toLocaleDateString("en-GB");
}

export function formatTime(dateStr: string, lang?: string): string {
  return new Date(dateStr).toLocaleTimeString(lang === "ar" ? "ar-EG" : "en-US", {
    hour: "2-digit",
    minute: "2-digit",
  });
}
```
NOTE: Check the actual implementations in the codebase before writing — some components use slightly different locale options (e.g., ReturnedChecksView formatCurrency takes `(amount, currency)` — that variant is component-specific and should remain inline or be a wrapper). The shared functions should match the most common pattern (the single-arg `formatCurrency(val: number)` used in ~15 places).

**jwt.ts** — Extract the JWT decode function from authSlice.ts:
```typescript
import type { UserRole } from "@/store/authSlice";

export function decodeJwt(token: string): { sub: string; role: UserRole; customer_id?: string } | null {
  try {
    const b64 = token.split(".")[1].replace(/-/g, "+").replace(/_/g, "/");
    const payload = JSON.parse(atob(b64));
    return { sub: payload.sub, role: payload.role as UserRole, customer_id: payload.customer_id };
  } catch {
    return null;
  }
}
```
NOTE: UserRole is defined in authSlice.ts. To avoid circular imports, either: (a) move UserRole type to jwt.ts and re-export from authSlice, or (b) define UserRole inline in jwt.ts and import it back in authSlice. Option (a) is cleaner — put the type in jwt.ts, re-export from authSlice for backward compat.

**product.ts** — Extract getProductName:
```typescript
import i18n from "@/i18n";

interface HasProductName {
  name_ar: string;
  name_en: string;
}

export function getProductName(product: HasProductName): string {
  return i18n.language === "ar" ? product.name_ar : product.name_en;
}
```
This replaces the inline `i18n.language === "ar" ? p.name_ar : p.name_en` pattern found in ~10 places. Check if any component uses `useTranslation()` i18n instance vs the global `i18n` import — the global import is fine since language doesn't change within a render.
