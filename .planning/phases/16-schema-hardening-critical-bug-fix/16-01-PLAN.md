    In `backend/app/services/payment_service.py`, in the `return_check` method, the original check transaction's status is set to `TransactionStatus.Returned` in memory (line 144) but never persisted. The method only calls `create_many` for the new return transaction and never calls `update` on the original check.

    Fix: Add `await self._transactions.update(check_txn)` after setting the status and before creating the return transaction. The updated method section should be:

    ```python
    check_txn.status = TransactionStatus.Returned
    await self._transactions.update(check_txn)  # <-- ADD THIS LINE

    original_amount = abs(check_txn.amount)
    return_txn = Transaction(
        # ... rest unchanged
    )
    ```

    Place the update call right after `check_txn.status = TransactionStatus.Returned` (current line 144) and before the `original_amount = abs(...)` line. The `update` method on the transaction repository calls `commit()` and `refresh()`, which will persist the status change.

    Note: The subsequent `create_many([return_txn])` call also commits, so both the status update and new transaction will be persisted. If you want a single atomic commit, you could alternatively move the update call to just before `create_many` — but the current pattern (update then create_many) is consistent with how `deposit_check` and `undeposit_check` work in this same service.
